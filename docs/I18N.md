# Internationalization (i18n) — Architecture & Decisions

Status: **Foundation landed** (PUP-475). Epic: **PUP-473**.

This document records the architectural decisions behind Code Puppy's i18n
system and gives translators a quickstart. It is the living answer to the
"Open Questions" section of PUP-473.

---

## 1. Initial target locales

The first languages we translate to are:

| Locale   | Catalog file              | Notes                                    |
|----------|---------------------------|------------------------------------------|
| `en-US`  | `locales/en-US.json`      | Source / default. Never missing a key.   |
| `es`     | `locales/es.json`         | Generic (Iberian-leaning) Spanish base.  |
| `es-419` | `locales/es-419.json`     | **Latin American Spanish umbrella** — the standard shared across Central & South America and the Caribbean. |
| `fr-CA`  | `locales/fr-CA.json`      | Canadian French specifically.            |

### Central & South American Spanish dialects

Per CLDR *parent-locale* rules, every Central/South American country's Spanish
falls back through the `es-419` umbrella **before** generic `es`:

```
es-AR -> es-419 -> es -> en-US     (Argentina, voseo)
es-MX -> es-419 -> es -> en-US     (Mexico)
es-CO -> es-419 -> es -> en-US     (Colombia)
...
es-ES -> es -> en-US               (Spain does NOT route through es-419)
```

This is implemented via `PARENT_LOCALES` in `code_puppy/i18n/locale.py`. To
add a specific country dialect (e.g. Argentine voseo), create
`locales/es-AR.json` containing **only** the strings that differ from
`es-419` — everything else is inherited automatically. Adding a new country
to the umbrella is a one-line change to `PARENT_LOCALES`.

#### Country dialect files shipped (stubs — pending native review)

These four ship as **empty override files** today. They are wired into the
fallback chain and fully inherit `es-419`; a native speaker fills in only the
keys that genuinely differ. **Do not** copy the whole `es-419` catalog into
them — only add divergent strings.

| Locale  | Country   | Notable divergence to watch for                 | Status            |
|---------|-----------|-------------------------------------------------|-------------------|
| `es-MX` | Mexico    | Mexican vocabulary; broad LatAm-neutral base    | Stub, needs review |
| `es-AR` | Argentina | **Voseo** (“vos”/“tenés” vs “tú”/“tienes”)         | Stub, needs review |
| `es-CO` | Colombia  | Relatively neutral; “usted”-leaning register     | Stub, needs review |
| `es-CL` | Chile     | Chilean vocabulary/register                      | Stub, needs review |

Call `code_puppy.i18n.available_locales()` for the live list of catalogs.

Adding a regional variant later is additive: drop a new `<locale>.json` next
to these and it participates in the fallback chain automatically. You only
need to include the keys that *differ* from the parent.

## 2. Decisions (ADR-lite)

### D1 — Catalog format: **JSON** for the CLI foundation

**Decision:** message catalogs are flat JSON keyed by dotted translation IDs.
A value is either a string or a plural-forms object.

**Why:** zero new dependencies for the day-one foundation, trivially
diffable, and the same format the frontend JS can consume. gettext/PO has
richer tooling (pybabel) but is JS-hostile and heavier than the foundation
needs.

**Consequence / later:** the *string-extraction* workstreams (PUP-480 public,
PUP-481 private, PUP-477 frontend) may layer Babel/pybabel on top for
extraction + compilation without changing this runtime contract — Babel can
emit JSON. The Python frontend (PUP-478) will use `jinja2.ext.i18n`; whether
its catalogs are PO or JSON is that story's call, but the CLI stays JSON.

### D2 — Missing keys **never raise**

A key absent from every catalog in the fallback chain renders as the **key
itself** (`nope.not.here`). This guarantees the app never crashes on a
translation gap, and an untranslated string is visually obvious on screen.

### D3 — Fallback chain, most-specific-first, default last

`zh-Hans-CN` → `zh-Hans` → `zh` → `en-US`. The default (`en-US`) is always
appended, so a key present only in the source catalog still resolves in any
locale.

### D4 — Interpolation is `{param}`-style and forgiving

We use `str.format_map` with a safe mapping. A missing `{param}` leaves the
placeholder intact instead of raising. Placeholders are **never** part of the
translatable text semantics — translators reorder them freely.

### D5 — Backend localization model: **code + params, localize in the CLI**

For `puppy-backend` (PUP-479): the backend returns **stable machine error
codes + params** (e.g. the `QuotaExceeded` struct), and the CLI localizes at
the edge. The backend does **not** bake N languages into the Rust binary.

Rationale: the struct-first design already separates data from prose, and
CLI-side localization avoids adding an `Accept-Language`/`x-puppy-locale`
round-trip for the common path. The only mandatory backend prose is the
`fake_llm_response` policy blurbs (which are returned as assistant turns and
printed verbatim) — those get a small dedicated catalog. **Before** any of
this, consolidate the duplicated `rate_limit_error` builder
(`openai_responses.rs` vs `llm_shim.rs`) so we don't translate one and orphan
its twin.

### D6 — Model-facing system prompts are **out of scope** for UI i18n

Translating agent/model system prompts changes LLM behavior and is a separate
track, not part of UI localization. Do not run them through the string
extractor.

### D7 — The `walmart_specific` plugin corpus is its **own epic** (PUP-482)

External translators cannot access the private plugin, so its ~727 strings
live behind a different security/translation boundary and are decoupled from
the public catalogs.

## 3. How the runtime resolves a locale

Precedence, highest first (see `code_puppy/i18n/locale.py`):

1. `CODE_PUPPY_LOCALE` env var (explicit override)
2. POSIX vars: `LC_ALL` → `LC_MESSAGES` → `LANG` → `LANGUAGE`
3. The persisted `locale` config key (`puppy.cfg`) — set via `/set locale`
4. Default: `en-US`

At boot, resolve once:

```python
from code_puppy.i18n import use_detected_locale
from code_puppy.config import get_value

use_detected_locale(get_value("locale"))
```

## 4. Call-site usage

```python
from code_puppy.i18n import t, ngettext, lazy

emit_info(t("startup.welcome", name=owner))          # simple
emit_info(ngettext("files.deleted", count=n))        # plural-aware
emit_info(lazy("startup.ready"))                      # resolved at render
```

`lazy(...)` returns a deferred translation resolved when the message bus calls
`str()` on it — use it for strings produced before the locale is finalized.
The single emit choke point (`messaging/message_queue.py::emit_message`)
resolves `LazyTranslation` automatically; plain strings pass through untouched.

Locale-aware formatting:

```python
from code_puppy.i18n import format_number, format_datetime
format_number(1234567)                # "1,234,567" (en) / "1.234.567" (de)
format_datetime(dt)                   # ISO for now; Babel-backed later
```

## 5. Translator quickstart

To add or update a translation:

1. Open the base catalog `code_puppy/i18n/locales/en-US.json` to see the keys
   and English source strings.
2. Create/edit `code_puppy/i18n/locales/<your-locale>.json`. Include only the
   keys you're translating; anything you omit falls back automatically.
3. Keep `{placeholders}` **exactly as written** — reorder them for grammar,
   but don't rename or remove them.
4. For plurals, provide the categories your language uses. Only `other` is
   required; add `one`/`few`/`many`/etc. as needed:
   ```json
   "files.deleted": { "one": "…", "other": "…" }
   ```
   Supported plural rules live in `code_puppy/i18n/plurals.py`. If your
   language isn't listed there yet, it defaults to the English one/other rule
   — file a note so we can add the correct CLDR rule.
5. Run the tests: `pytest tests/i18n/`.

### Pseudolocalization (catching missing translations)

Set the locale to `en-XA` to render every string in an accented, padded,
bracketed form: `⟦Ŵéĺçöḿé ţö Çöđé Þüþþý⟧`. Any text that shows up **without**
brackets is a hardcoded string that skipped the catalog — a bug. This is what
the pseudolocale CI check (PUP-476) will assert on.

## 6. Where things live

```
code_puppy/i18n/
├── __init__.py      # public API: t, ngettext, lazy, format_*, set_locale…
├── locale.py        # detection, normalization, fallback chain
├── catalog.py       # JSON loader, merge, missing-key lookup
├── translate.py     # Translator singleton, t(), ngettext(), LazyTranslation
├── plurals.py       # CLDR-lite plural category selection
├── formats.py       # locale-aware number/date formatting (stdlib for now)
├── pseudo.py        # pseudolocalization transform
└── locales/
    ├── en-US.json   # source / default
    ├── es.json      # Spanish (generic / Iberian base)
    ├── es-419.json  # Latin American Spanish umbrella (Central & South America)
    ├── es-MX.json   # Mexico       (stub -> inherits es-419)
    ├── es-AR.json   # Argentina    (stub -> inherits es-419)
    ├── es-CO.json   # Colombia     (stub -> inherits es-419)
    ├── es-CL.json   # Chile        (stub -> inherits es-419)
    └── fr-CA.json   # Canadian French
```
